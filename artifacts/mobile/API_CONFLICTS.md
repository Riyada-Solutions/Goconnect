# API Conflicts — v2 Postman Collection vs. Mobile App Code

Generated by comparing the Postman collection
`CareConnect_Nurse_—_Backend_API_(v2_—_mobile-contract_bodies)`
against the mobile app code under `data/`.

Severity legend: 🔴 breaks calls · 🟠 wrong field names (silent data loss) · 🟡 read-side mismatch · 🟢 minor / extra

---

## 1. Auth & Profile

### 1.1 🔴 `POST /auth/register` — body shape changed completely
**Server (v2):**
```json
{ "registerCode": "ABC123", "phone": "+966500000099", "fullName": "New Nurse", "email": "newnurse@example.com" }
```
**App:** sends `{ registerCode, phone, username, name, email, password, password_confirmation }`.
**Differences:** `name` → `fullName`; `username`, `password`, `password_confirmation` removed (registration no longer takes a password — flow now relies on OTP).
**Action:** **Fixed.** `RegisterRequest` model + `auth_repository.register` updated.

### 1.2 🔴 `POST /auth/verify-otp` — `email` → `identifier`
**Server:** `{ purpose, identifier, otp }`
**App:** previously sent `{ purpose, email, otp }`.
**Action:** **Fixed.** `VerifyOtpRequest` now requires `identifier`; the OTP screen passes the email under that field. The repo posts the typed body verbatim — every caller must use `identifier`.

### 1.3 🟢 All other auth endpoints match
`login`, `me`, `update profile`, `forgot-password`, `reset-password`, `change-password`, `logout`, `delete-account`, `device-token`, `verify-face` — request and response shapes match.

## 2. Authorization (`GET /me/rules`)

### 2.1 🟠 Action-key vocabulary differs
**Server returns bundle-level keys** (one per form):
`submit_flowsheet`, `submit_progress_notes`, `submit_allergies`, `submit_social_assessment`, `submit_incidents`, `submit_blood_sugar`, `submit_visual_triage_checklist`, etc.

**App's `RuleAction` type uses per-section keys** like
`submit_flow_sheet_outside_dialysis`, `submit_flow_sheet_pre_treatment_vitals`, …
and split keys for `submit_doctor_progress_note` / `submit_social_worker_progress_note`.

**Impact:** every per-section gate in the UI evaluates to `false` against a real backend response → flow-sheet save buttons would be hidden.
**Recommended action (decision required):**
- **Option A (preferred):** ask backend to emit the per-section keys we already have in `data/models/rules.ts`.
- **Option B:** collapse the app keys to the server bundles (`submit_flowsheet`, `submit_progress_notes`) and remove per-section gating.
**Status:** **NOT changed yet — needs product/backend decision.**

## 4. Patients

### 4.1 🟡 `GET /patients/{id}/alerts` — response is an **object**, not an array
**Server:** `{ allergies: [...], contamination: [...], instructions, isolation }` (matches `PatientAlertSummary`).
**App `getPatientAlerts`** returns `PatientAlert[]` and does `Array.isArray(payload) ? payload : []` → it always returned `[]`.
**Action:** **Fixed.** New `getPatientAlertSummary` added; legacy `getPatientAlerts` is now a deprecated stub returning `[]`.

> **Note — the visit-detail screen does not call this endpoint.** It reads `record.patientAlerts` (the embedded summary on the visit response) directly. `Visit.patientAlerts` is now typed as `PatientAlertSummary | PatientAlert[] | PatientAlert | null` so both v2 and legacy mock shapes type-check; new screens that need a stand-alone alert fetch should call `getPatientAlertSummary` instead.

### 4.2 🟢 List + detail match.

## 7. Visits — read & status

### 7.1 🟢 Match. `GET /visits`, `GET /visits/{id}`, `POST /visits/{id}/start`, `/end`, `/procedure-times` align.

### 7.2 🟡 `forms.flowsheet[*].value` keys
The visit-detail response stores flow-sheet sections under the **new** snake_case keys (`pre_treatment_vitals`, `fall_risk`, `nursing_actions`, `dialysis_parameters`, `medications`, `post_treatment`). The app's read mapper (`mapFlowSheetFromApi` in `data/visit_repository.ts`) used the **old** keys (`pre_treatment_vital`, `fall_risk_assessment`, …).
**Action:** **Fixed.** Mapper updated to read new keys (with old keys as fallback).

## 8. Visit forms — generic endpoint

All section sends were going to `POST /visits/{id}/forms/flowsheet` under wrong top-level keys.

### 8.1 🔴 Section-key map (FLOWSHEET_SECTION_KEY)
| Section slug             | Old (app, wrong)         | New (server) |
|--------------------------|--------------------------|--------------|
| `pre-treatment-vitals`   | `pre_treatment_vital`    | `pre_treatment_vitals` |
| `fall-risk`              | `fall_risk_assessment`   | `fall_risk` |
| `nursing-actions`        | `nursing_action`         | `nursing_actions` |
| `dialysis-parameters`    | `hemodialysis`           | `dialysis_parameters` |
| `medications`            | `dialysis_medications`   | `medications` |
| `post-treatment`         | `post_assessment`        | `post_treatment` |

Other keys (`outside_dialysis`, `machines`, `pain_assessment`, `alarms_test`, `intake_output`, `car`, `access`, `dialysate`, `anticoagulation`) were already correct.
**Action:** **Fixed** in `FLOWSHEET_SECTION_KEY`.

### 8.2 🔴 Post-treatment body — shape **and** key wrong
**Server v2:** posts `{ post_treatment: { postTx: { …camelCase fields… }, patientSignature?, nurseSignature? } }`. Per-section pattern for the multipart upload mirrors the JSON shape (signatures as files; `data` JSON part contains the same payload).
**App (old):** posted `{ post_assessment: <serializePostTx flat snake_case> }`.
**Action:** **Fixed.** `submitFlowSheetPostTreatment` now wraps the camelCase `postTx` under `post_treatment`. The legacy `serializePostTx` snake_case translator was removed.

### 8.3 🔴 Nursing progress note: `notes` → `note`
**Server:** `{ "note": "..." }`
**App:** sent `{ "notes": "..." }`
**Action:** **Fixed.**

### 8.4 🔴 Doctor progress note: completely different shape
**Server:** `{ type: "doctor", note, isAddendum, parentNoteId }` — also supports `type: "doctor"` with `isAddendum: true` for addenda.
**App:** sent `{ type: "in_visit", notes, addenda: [...] }`.
**Action:** **Fixed.**

### 8.5 🔴 Social worker progress note: shape differs
**Server:** `{ type: "social_worker", note, location: "in_center" | "on_call" }`
**App:** sent `{ type: <derived>, notes, addenda: [] }` (no location, derived type).
**Action:** **Fixed.**

### 8.6 🟠 SARI screening: `visitId` should not be in body
**Server:** body is `{ addressographPatientName, dateTime, sariFeatures, exposureCriteria, actions }` — `visitId` is in the URL only.
**App:** spread the entire payload (including `visitId`) into the body.
**Action:** **Fixed** — `visitId` is now stripped before posting.

### 8.7 🟢 Refusal & referral (multipart) — no field-name conflicts detected. Existing multipart layout still applies.

### 8.8 🟢 Inventory usage — match (`{ itemId, quantity, notes }`).

## Summary of code changes applied in this pass

1. `data/models/auth.ts` — `RegisterRequest` now `{ registerCode?, phone, fullName, email }`; `VerifyOtpRequest` uses `identifier` instead of `email`.
2. `data/auth_repository.ts` — body for `verifyOtp` adjusted (no behavior change for callers).
3. `data/visit_repository.ts`:
   - `FLOWSHEET_SECTION_KEY` updated for the 6 renamed sections.
   - `mapFlowSheetFromApi` updated to read v2 keys (with v1 fallbacks).
   - `submitFlowSheetPostTreatment` rewritten to send the camelCase `postTx` under `post_treatment` (JSON or `data`-part of multipart).
   - `submitNursingProgressNote` body field `notes` → `note`.
   - `submitDoctorProgressNote` body now `{ type: "doctor", note, isAddendum, parentNoteId? }`.
   - `submitSocialWorkerProgressNote` body now `{ type: "social_worker", note, location }`.
   - `submitSariScreening` strips `visitId` before sending.
4. `data/patient_repository.ts` — added `getPatientAlertSummary` returning the new object shape; legacy `getPatientAlerts` kept (returns `[]`) and marked deprecated.
